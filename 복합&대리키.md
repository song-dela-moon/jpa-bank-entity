# Serializable와 복합키(Composite Key) vs 인조식별자(Surrogate Key)

## 1. Serializable이란?

`Serializable`은 Java의 **마커 인터페이스(Marker Interface)** 로, 구현할 메서드가 없이 단지 선언만으로
JVM에게 **"이 객체는 바이트 스트림으로 변환(직렬화)할 수 있다"** 고 알려주는 역할을 합니다.

```java
public interface Serializable {
    // 메서드가 하나도 없음 — 선언만으로 '표시(마킹)'
}
```

### 직렬화(Serialization) / 역직렬화(Deserialization)

```
[직렬화]   Java 객체  ──►  바이트 스트림(byte[])  ──►  파일, 네트워크, 캐시 등
[역직렬화] 파일, 네트워크, 캐시 등  ──►  바이트 스트림(byte[])  ──►  Java 객체
```

- **직렬화** = 객체를 전송·저장 가능한 형태로 변환 (택배 포장)
- **역직렬화** = 바이트에서 원래 객체로 복원 (택배 개봉)

### 사용처

| 사용처 | 설명 |
|---|---|
| 파일 저장 | 객체를 파일로 저장하고 나중에 복원 |
| 네트워크 전송 | 객체를 다른 서버로 전송 (RMI, 소켓 등) |
| 세션 저장 | HttpSession에 저장된 객체를 디스크에 백업 |
| 캐시 | Hibernate 2차 캐시, Redis 등에 객체 저장 |
| JPA 복합키 | `@EmbeddedId` 클래스는 반드시 Serializable 필요 |

## 2. JPA에서 복합키 클래스에 Serializable이 필요한 이유

JPA 스펙(JSR 338)에서 **복합키 클래스는 반드시 `Serializable`을 구현해야 한다**고 명시하고 있습니다.

### 직렬화 대상은 PK 객체뿐

```
직렬화 대상: AccountCustomerId (복합키 객체 자체)
    ├── accountId (Long) → 단순한 숫자 값
    └── customerId (Long) → 단순한 숫자 값

❌ Account 엔티티 전체를 직렬화하는 것이 아님
❌ Customer 엔티티 전체를 직렬화하는 것이 아님
```

### 인조식별자 방식에서는 선언이 불필요한 이유

`Long`, `Integer`, `String` 등 Java 기본 래퍼 타입은 **이미 `Serializable`을 구현**하고 있기 때문입니다.

```java
public final class Long extends Number implements Serializable { ... }
```

> **캐시 키 저장 시 직렬화가 필요한 건 모든 PK 타입 동일.**
> 단지 `Long`은 이미 구현되어 있어 신경 쓸 필요가 없고, 커스텀 복합키 클래스는 직접 선언이 필요할 뿐.

| PK 타입 | Serializable 구현 여부 | 직접 선언 필요? |
|---|---|---|
| `Long` | ✅ Java가 이미 구현 | ❌ |
| `Integer` | ✅ Java가 이미 구현 | ❌ |
| `String` | ✅ Java가 이미 구현 | ❌ |
| `AccountCustomerId` (커스텀) | ❌ 선언 안 하면 불가 | ✅ |

## 3. 복합키가 별로인 진짜 이유

> **직렬화 리소스 차이(Long 2개 vs 1개 = 16바이트 vs 8바이트)는 거의 무시할 수준.**
> 복합키가 별로인 이유는 직렬화 때문이 아니라 아래의 이유들 때문이다.

### 3-1. DB 성능 — 조인과 인덱스

```sql
-- 인조식별자: FK 1개 → 조인 조건 단순
SELECT * FROM Order o
JOIN Account_Customer_Surrogate acs 
  ON o.account_customer_id = acs.Account_Customer_id

-- 복합키: FK 2개 → 조인 조건 복잡, 인덱스도 2개 컬럼 필요
SELECT * FROM Order o
JOIN Account_Customer_Composite acc 
  ON o.account_id = acc.Account_id 
  AND o.customer_id = acc.Customer_id
```

- FK로 참조 시 **컬럼이 2개씩** 필요
- 테이블이 많아질수록 조인 조건이 **기하급수적으로 복잡**해짐
- 복합 인덱스는 단일 컬럼 인덱스보다 크기가 크고 B-Tree 비교 연산도 증가

### 3-2. 코드 복잡도 — CRUD가 전부 불편

```java
// 인조식별자
em.find(AccountCustomerSurrogate.class, 1L);

// 복합키 — 매번 ID 객체를 생성해야 함
em.find(AccountCustomerComposite.class, new AccountCustomerId(1L, 2L));
```

```
// 인조식별자: REST API도 깔끔
GET /account-customer/1

// 복합키: URL이 복잡해짐
GET /account-customer?accountId=1&customerId=2
```

### 3-3. 확장성 — 키 변경이 치명적

복합키에 컬럼을 추가하면, 이 키를 FK로 참조하는 **모든 테이블**에도 해당 컬럼을 추가해야 한다.

```java
// Before
public class AccountCustomerId implements Serializable {
    private Long accountId;
    private Long customerId;
}

// After — branchId 추가 시
public class AccountCustomerId implements Serializable {
    private Long accountId;
    private Long customerId;
    private Long branchId;    // ← FK 참조하는 모든 테이블도 수정 필요!
}
```

인조식별자는 PK가 `Long id` 그대로이므로 **FK를 참조하는 테이블은 아무 변경 없음**.

### 3-4. equals()/hashCode() 관리 부담

복합키 클래스는 반드시 `equals()`와 `hashCode()`를 올바르게 구현해야 한다.
잘못 구현하면 캐시 미스, 엔티티 중복 등 **디버깅하기 어려운 버그**가 발생할 수 있다.

---

## 4. 종합 비교

| 비교 항목 | 복합키 (`@EmbeddedId`) | 인조식별자 (`@Id Long`) |
|---|---|---|
| 직렬화 성능 | 거의 차이 없음 | 거의 차이 없음 |
| **조인 복잡도** | ❌ FK 컬럼 2개+ | ✅ FK 컬럼 1개 |
| **CRUD 코드** | ❌ ID 객체 매번 생성 | ✅ Long 하나 |
| **API 설계** | ❌ 복잡한 URL/파라미터 | ✅ 깔끔 |
| **확장성** | ❌ 키 변경 시 연쇄 수정 | ✅ PK 변경 없음 |
| **데이터 무결성** | ✅ DB가 자동 보장 | ⚠️ UniqueConstraint 별도 필요 |

> **결론**: 복합키가 별로인 이유는 직렬화 때문이 아니라, 주로 **조인 복잡도, 코드 불편함, 확장성 부족** 때문이다.
> 반면 복합키의 확실한 장점은 **DB 레벨에서 중복을 자동 차단**해준다는 점이다.
