해당 이미지는 `Question`과 `Answer`를 관리하는 테이블이다.

1. 나는 JPA를 고려하기 앞서서 데이터베이스의 관계 모델링 관점으로 보았을 때 복합키를 사용하면 좋다고 생각하였다.
2. 팀원은 JPA를 사용하려면 대리키를 사용하는 것이 관리면에서 훨씬 유용하다는 의견을 제시했다.
3. 어떻게하면 JPA에 맞게 테이블을 리펙토링 할 수 있을까?

# 1. 복합키(Embedded Key) 사용

`member_id`, `club_id`, `question_id`를 `@EmbeddedId`를 이용해 하나의 기본 키로 묶는 방식이다.

처음 DB를 설계할 땐 `Answer` 테이블에는 Primary Key가 존재하지 않아도 된다고 생각했다.

왜냐하면 지원서에서 Answer의 레코드는 무조건 Question이 존재할 경우만 생성될 수 있으니 말이다.

### 장점

- **PK 기반의 물리적 정렬** ⭐***(내가 가장 중요하게 생각했던 Point)***
    - InnoDB 스토리지 엔진은 **Primary Key를 클러스터형 인덱스로 사용한다**. 
    따라서 **데이터 자체가 기본 키 순서에 따라 디스크에 물리적으로 정렬되어 저장된다**. 
    이 구조는 특정 범위의 데이터를 조회*(페이징)*할 때 디스크 I/O를 최소화하여 뛰어난 성능을 보임.
- **자연키를 통한 데이터 무결성 보장**
    - 키 자체가 '(멤버, 동아리, 질문)'의 조합이므로, 한 사용자가 특정 질문에 중복 답변하는 것을 **DB 레벨에서 원천적으로 차단**된다.
- **명확한 논리적 표현**
    - 키(Key)만 봐도 데이터가 어떤 의미인지 명확하게 드러난다.
- **공간 효율성**
    - 식별만을 위한 별도의 인공적인 컬럼이 필요 없어 테이블 구조가 깔끔하다.

### 단점

- **JPA 사용의 복잡성 증가**
    - 복합 키를 다루기 위해 별도의 `IdClass`(@EmbeddedId)를 만들어야 하고, `equals`와 `hashCode`를 직접 오버라이딩해야 하는 등 **코드가 복잡해지고 관리 포인트가 늘어난다.**
- **테이블 수정 및 관계의 어려움**
    - 다른 테이블에서 `Answer`를 참조(FK)할 때, 3개의 컬럼을 모두 포함해야 하므로 관계가 복잡해지고 쿼리가 길어진다.
- **쿼리 메소드의 한계**
    - Spring Data JPA가 제공하는 `findById` 같은 기본적인 쿼리 메소드를 직관적으로 사용하기 어렵다.

# 2. 대리키(Surrogate Key) 사용

우리가 프로젝트에서 선택한 **DB Connection**은 **Hibernate를 이용한 JPA**이다.

비즈니스적인 의미는 없지만, 레코드를 식별하기 위한 고유 ID(`answer_id`)를 기본 키로 사용하는 방식이다. 

JPA의 `@Id`와 `@GeneratedValue`를 함께 사용하는 가장 일반적인 방식이다.

### 장점

- **JPA/ORM 친화적**
    - **가장 큰 장점!
    JPA는 이 방식에 최적화되어 있어, 엔티티 조회, 수정, 삭제, 관계 매핑 등 모든 과정이 매우 단순하고 직관적으로 변한다.**
- **단순하고 명확한 참조**
    - 다른 테이블에서 특정 답변을 참조할 때 `answer_id` 하나만 참조하면 되므로 관계 맺기가 매우 쉽다.
- **수정의 자유로움**
    - 만약 복합 키의 일부가 변경되는 상황이 발생하더라도, 대리 키는 아무런 영향을 받지 않아 **변경에 자유롭다.**

### 단점

- **데이터 무결성은 별도 보장**
    - `answer_id`는 레코드의 고유성만 보장할 뿐이다.
    - **"한 멤버가 한 동아리의 한 질문에 하나의 답변만 할 수 있다"**는 비즈니스 규칙을 보장해주지는 못한다.
    - 따라서 이 규칙을 지키려면 **테이블에 `UNIQUE` 제약 조건을 추가로 설정**해야한다. (`UNIQUE (member_id, club_id, question_id)`)

# 최종 결정 및 결론

이 문제는  데이터 무결성을 중시하는 **데이터베이스 중심 설계**와, 개발 생산성을 강조하는 **ORM 중심 설계** 사이의 고전적인 트레이드 오프입니다.

또한 프로젝트를 시작하며 겪은 최초의 트레이드 오프이기도 합니다.

저희는 여러 논의 끝에 팀원이 제시한 "**2번 - 대리 키(@GeneratedValue)를 사용"방식**을 최종적으로 채택했습니다.

가장 결정적인 이유는 **"우리는 JPA를 사용하고 있기 때문"**입니다. 
JPA를 도입한 이유는 객체 지향적으로 데이터를 관리하고, 반복적인 SQL 작업을 줄여 **개발 생산성을 높이기 위함**입니다. 
복합 키를 사용하면 이러한 JPA의 장점을 상당 부분 포기해야만 합니다.

또한, 대리 키의 유일한 단점인 '비즈니스 키 무결성'은 **테이블 레벨에서 `UNIQUE` 제약 조건을 추가**함으로써 데이터베이스의 힘을 빌려 보완할 수 있었습니다. 

결국, **JPA와의 완벽한 호환성 및 개발 생산성이라는 큰 이점**을 취하면서도, **`UNIQUE` 제약 조건으로 단점을 해결**할 수 있는 대리 키 방식이 우리 프로젝트에 더 합리적인 선택이라고 결론 내렸습니다.

막상 정리해보니 지금 이걸 보는 면접관님이 정말 기초적인 문제에 대해 써놨다고 생각이 들까봐 조금 부끄러웠고 그냥 지울까? 라는 생각도 들었습니다..

**하지만 이론으로 공부하던 내용과 현실은 다르다는 걸 느끼고 팀원에 말에 경청하는 계기가 되었기 때문에** 작성해본 문제 해결입니다.
